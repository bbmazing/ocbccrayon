A tool is just a function

No proprietary magic — you write the function

# a tool is a normal function
def get_balance(account_id):
  return db.lookup(account_id)
  • A tool = a normal function (DB query, API call, calc)...
  • ...plus a description + input schema the model can read.
  • The model decides when to call it; your code runs it.

Anatomy of a tool

Name, description, input schema

# what the model actually sees
{
  "name": "get_account_balance",
  "description": "Balance for a
   customer's account, in SGD.",
  "input_schema": {
   "account_id": string
  }
}
  • Name — what to call it.
  • Description — what it does & when to use it.
  • Input schema — typed arguments the model must supply.
  • The model never sees your code — this is the prompt for the tool.

Writing good tools

Easy for the model

"get_account_balance"1–2 argsdistinct purpose

Why planning matters

Real tasks have many dependent steps

  • One-step questions need no plan — just answer.
  • Real tasks have dependencies: do A before B before C.
  • Without a plan: the agent wanders, repeats, or stops early.
  • Planning = sequence steps + know when you're done.

Pattern 1 · ReAct

Reason + Act, one step at a time

ReActSTEP BY STEP
Reason
Act
Observe
  • Think out loud → one action → observe → repeat.
  • Plans one step at a time on real feedback.
  • Readable reasoning trace — easy to debug.
  • The sensible default for most agents.

Failure modes

What goes wrong in production

Runaway loops

Stuck repeating actions — a real case burned ~US$47K in API costs.

Prompt injection

Malicious text in a doc/page hijacks the agent's instructions.

Hallucinated calls

Invents a tool or fakes arguments that don't exist.

Compounding errors

A wrong early step quietly corrupts every step after.

⚠ Autonomy without limits is a liability — hence guardrails, next.

Guardrails

The agent proposes; a human disposes

Least privilege

Only the tools it truly needs — nothing more.

Hard limits

Caps on steps, spend and time — no runaway loops.

Human-in-the-loop

A person approves money-moving or irreversible actions.

Validation

Check tool inputs/outputs; sandbox any code execution.

In a regulated bank these are not optional.
